# McDonald 订单管理系统 - 需求规格说明书

**文档编号**: REQ-001
**版本**: 1.0
**创建日期**: 2026-06-25
**作者**: Product Owner

---

## 一、文档目的

本文档用大白话解释 McDonald 自动化订单管理系统的所有需求，让开发团队、测试团队和业务方都能看懂。

---

## 二、业务背景

McDonald's 在疫情期间生意太好，人手不够了！他们买了一批烹饪机器人来处理订单。作为软件工程师，你的任务是做一个"订单控制器"来管理这些机器人和订单。

---

## 三、用户角色

| 角色 | 说明 |
|------|------|
| 普通顾客 (Normal Customer) | 普通下单用户，订单排队等处理 |
| VIP 会员 (VIP Customer) | 尊贵会员，订单优先处理 |
| 餐厅经理 (Manager) | 可以增加/减少机器人数量 |
| 烹饪机器人 (Bot) | 自动处理订单，每单需要10秒 |

---

## 四、核心需求（7条）

### 需求 1：普通订单
**操作**：点击 "New Normal Order"（新建普通订单）

**期望结果**：
- 系统自动生成一个唯一编号的订单（比如 #1, #2, #3...）
- 订单进入 "PENDING"（待处理）区域
- 订单排在队列最后面

**大白话**：来了个普通顾客下单，订单排队等着机器人来做。

---

### 需求 2：VIP 订单
**操作**：点击 "New VIP Order"（新建 VIP 订单）

**期望结果**：
- 系统自动生成一个唯一编号的订单
- 订单进入 "PENDING"（待处理）区域
- 订单排在所有普通订单前面，但在所有其他 VIP 订单后面

**大白话**：VIP 顾客来了！虽然要排队，但排在所有普通人前面。如果前面有其他 VIP，就跟在那个 VIP 后面。

---

### 需求 3：订单编号规则
**规则**：
- 订单编号必须唯一，不能重复
- 订单编号必须递增（1 → 2 → 3 → ...）
- 不管是普通订单还是 VIP 订单，都用同一个编号池

**大白话**：订单编号就像发票号，1号、2号、3号...一个萝卜一个坑，不能有两个#1。

---

### 需求 4：添加机器人
**操作**：点击 "+ Bot"（增加机器人）

**期望结果**：
- 创建一台新的机器人，立即开始工作
- 机器人从 PENDING 区域取走排在最前面的订单（VIP 优先）
- 机器人需要 10 秒来完成这个订单
- 10 秒后，订单从 PENDING 移到 "COMPLETE"（已完成）区域
- 如果 PENDING 还有订单，机器人继续取下一个

**大白话**：老板招了个新员工！新员工一来就开工，从队伍最前面拿一单开始做。做完了（10秒），把完成的单放到"已完成"区域，然后继续做下一单。

---

### 需求 5：机器人空闲状态
**条件**：PENDING 区域没有任何订单

**期望结果**：
- 所有机器人都处于 "IDLE"（空闲）状态
- 等待新订单到来

**大白话**：没单子了！所有员工都在休息玩手机，等新顾客来点单。

---

### 需求 6：移除机器人
**操作**：点击 "- Bot"（减少机器人）

**期望结果**：
- 移除最近创建的那台机器人
- 如果机器人正在处理订单，立即停止
- 那笔被中断的订单放回 PENDING 区域
- 放回时保持原有的优先级（VIP 还是 VIP，普通还是普通）

**大白话**：老板开除最后招的那个员工！如果他正在做单，做了一半的单要放回队伍里。这半成品单还是 VIP 就是 VIP，是普通单就是普通单。

---

### 需求 7：无数据持久化
**规则**：所有数据都存在内存里，程序退出就丢失

**大白话**：系统就是个"临时工"，关电脑就没了。不需要存数据库，不用保存文件。

---

## 五、用户故事

### 故事 1：普通顾客下单
> 作为一个普通顾客，我希望下完单后看到订单在"待处理"区域，这样我知道我的单已经收到了。

**验收标准**：
- [ ] 点击"新建普通订单"按钮后，订单出现在 PENDING 区域
- [ ] 订单有唯一编号
- [ ] 订单编号比之前的订单大 1

---

### 故事 2：VIP 会员下单
> 作为一个 VIP 会员，我希望能比普通顾客先吃到汉堡，这样我觉得我的会员费没白交。

**验收标准**：
- [ ] 点击"新建 VIP 订单"后，订单出现在 PENDING 区域
- [ ] VIP 订单排在所有普通订单之前
- [ ] 如果已有其他 VIP 订单，新 VIP 排在已有 VIP 之后

---

### 故事 3：机器人处理订单
> 作为一个机器人，我知道一次只能做一单，做完一单再做下一单。

**验收标准**：
- [ ] 机器人创建后立即开始处理订单
- [ ] 每单处理时间为 10 秒
- [ ] 订单完成后自动移到 COMPLETE 区域
- [ ] 处理完当前订单后，机器人继续处理下一单（如果有）

---

### 故事 4：经理管理机器人
> 作为一个餐厅经理，我想随时增加或减少机器人数量，灵活应对订单高峰和低谷。

**验收标准**：
- [ ] 点击"+ Bot"能添加新机器人
- [ ] 点击"- Bot"能移除最新的机器人
- [ ] 移除机器人时，被中断的订单能正确放回队列

---

### 故事 5：机器人空闲等待
> 作为经理，我想知道什么时候机器人都在休息，这样我知道什么时候需要更多机器人。

**验收标准**：
- [ ] PENDING 为空时，机器人状态显示为 IDLE
- [ ] 新订单到来时，空闲机器人立即开始工作

---

## 六、需求分类

### P0 - 必须完成（核心功能）

| 编号 | 需求 | 验收条件 |
|------|------|----------|
| P0-1 | 创建普通订单 | 订单进入 PENDING，编号唯一递增 |
| P0-2 | 创建 VIP 订单 | VIP 排在普通订单之前 |
| P0-3 | 添加机器人 | 机器人立即开始处理订单 |
| P0-4 | 订单 10 秒完成 | 处理时间准确，订单移到 COMPLETE |

### P1 - 重要功能

| 编号 | 需求 | 验收条件 |
|------|------|----------|
| P1-1 | 机器人空闲状态 | PENDING 为空时显示 IDLE |
| P1-2 | 移除机器人 | 移除最新机器人，订单正确放回 |

### P2 - 可选功能

| 编号 | 需求 | 验收条件 |
|------|------|----------|
| P2-1 | 无数据持久化 | 所有操作在内存中完成 |

## 七、数据状态说明

### 订单状态

```
┌─────────────────────────────────────────────┐
│                  订单状态流程                  │
│                                             │
│    创建订单 ──→ PENDING ──→ COMPLETE         │
│                    ↑                        │
│                    │                        │
│              (被机器人取走                    │
│               正在处理中)                    │
└─────────────────────────────────────────────┘
```

### 机器人状态

```
┌─────────────────────────────────────────────┐
│                 机器人状态                    │
│                                             │
│   IDLE ←────── PENDING 为空                  │
│    │                                        │
│    │                                        │
│    ↓                                        │
│  PROCESSING ←─── PENDING 不为空              │
│    │                                        │
│    │                                        │
│    ↓                                        │
│ (10秒后完成，循环)                           │
└─────────────────────────────────────────────┘
```

## 八、PENDING 队列排序规则

VIP 订单优先，同类型订单按创建时间排序：

```
┌────────────────────────────────────────────────────────┐
│                    PENDING 队列                         │
├────────────────────────────────────────────────────────┤
│                                                        │
│   位置 1: VIP #3 (最早的 VIP)                          │
│   位置 2: VIP #5 (第二个 VIP)                          │
│   位置 3: VIP #7 (最新的 VIP)                          │
│   ─────────────────────────────────────                │
│   位置 4: Normal #1 (最早的普通)                        │
│   位置 5: Normal #2 (第二个普通)                        │
│   位置 6: Normal #4 (最新的普通)                        │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 九、边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 没有机器人，有订单 | 订单在 PENDING 等待 |
| 有机器人，没订单 | 机器人 IDLE，等待订单 |
| 既没机器人，也没订单 | 系统空闲，什么都不做 |
| 移除正在干活的机器人 | 订单放回 PENDING，保持原有优先级 |
| 同时创建多个订单/机器人 | 按指令顺序执行 |
| 订单编号到很大 | Go 的 int64 足够，放心用 |

---

## 十、成功标准

在开发完成时，以下条件必须全部满足：

- [ ] 可以创建普通订单，订单进入 PENDING
- [ ] 可以创建 VIP 订单，VIP 排在普通订单前面
- [ ] 订单编号唯一且递增
- [ ] 可以添加机器人，机器人开始处理订单
- [ ] 订单 10 秒后自动移到 COMPLETE
- [ ] PENDING 为空时机器人显示 IDLE
- [ ] 可以移除机器人，被中断的订单正确放回
- [ ] 无需数据持久化
- [ ] 输出包含 HH:MM:SS 格式时间戳

---

## 十一、术语表

| 术语 | 解释 |
|------|------|
| PENDING | 待处理区域，存放等待机器人处理的订单 |
| COMPLETE | 已完成区域，存放处理完毕的订单 |
| IDLE | 空闲状态，机器人没有工作时的状态 |
| VIP | 会员订单，有优先处理权 |
| Bot | 烹饪机器人，一次处理一个订单 |
| Normal | 普通订单，没有优先权 |
